# GreenLedger

Turn SME **invoice data into ESG sustainability signals** and a lender-ready **Financing Passport**.

> Course MVP for the FinTech: Business Models & Applications course (RSM, Erasmus University). Built on synthetic data, not production software.

## What it does

SMEs face growing pressure to disclose ESG data (from banks' Green Asset Ratio obligations and clients' CSRD reporting) but lack the infrastructure to collect it. GreenLedger derives ESG signals from data SMEs already produce (invoices) and turns them into a profile banks can act on.

The product works in two connected steps, both shown in the demo:

- **SME side:** upload invoice data → get an automated ESG scan and Green Credit Score → generate and share a Financing Passport.
- **Lender side:** open a shared Financing Passport → review the enriched borrower profile → make a lending decision.

## Key features

- **Invoice ingestion**: parse invoices into structured line items.
- **ESG classification**: embed line items with a transformer model, match them against EU Taxonomy categories by similarity, and ground each classification in regulation using retrieval-augmented generation (RAG).
- **Scoring**: aggregate classifications into a procurement sustainability score and a headline Green Credit Score with sector benchmarking.
- **Financing Passport**: a lender-ready bundle of ESG profile, financial-health metrics, requested financing amount, and Green Credit Score.
- **Data quality indicator**: a confidence tier shown alongside the Green Credit Score, reflecting how much classified data it's based on (never changes the score).
- **Lender comparison view**: the lender opens a roster of multiple SMEs (the real one that went through the pipeline, plus demo fixtures) side by side, picks one, and records an approve/decline decision with an amount and rate.
- **VSME report export**: a standardised ESG report the SME can share.

## Architecture

```
invoice data → ingestion → classification (embed + match + RAG) → scoring → passport → report
                                                                       │
                                            SME dashboard ◄────────────┘────────► Lender view
```

- **Backend:** FastAPI (Python) the pipeline above, exposed via SME and lender endpoints.
- **Frontend:** Streamlit SME dashboard and lender view.
- **Classification:** `sentence-transformers` embeddings + plain NumPy cosine-similarity search over precomputed EU Taxonomy category embeddings.

## Getting started

### Requirements
- Python 3.11+
- Backend dependencies in `pyproject.toml`, frontend dependencies in `frontend/requirements.txt`

### Run the app
The backend (FastAPI) and frontend (Streamlit) run as two separate processes.

```bash
git clone https://github.com/660579/FinTech_Joshua_Bas.git
cd FinTech_Joshua_Bas

python3 -m venv .venv
source .venv/bin/activate          # Windows: .venv\Scripts\activate

pip install -e .                          # backend
pip install -r frontend/requirements.txt  # frontend

python scripts/generate_synthetic_invoices.py                 # demo invoice data
python -m backend.app.services.classification.build_index     # precompute taxonomy embeddings

uvicorn backend.app.main:app --reload --reload-dir backend    # terminal 1
streamlit run frontend/app.py                                 # terminal 2
```

The app opens at http://localhost:8501, talking to the backend at
http://localhost:8000. If the backend isn't running or isn't reachable, every
screen falls back to mock data automatically and shows a warning. The demo
never crashes, but you'll want the backend up for real classification
results. RAG rationale text works without any API key (it falls back to a
templated sentence); set `GROQ_API_KEY` in a `.env` file (see
`.env.example`) for real LLM-generated rationales.

### Demo data

The MVP runs entirely on **synthetic invoice data** generated by `scripts/generate_synthetic_invoices.py`. No real bank, PSD2, or accounting-software connections are used. In a production setting these would be replaced by open-banking (AISP/PSD2) and accounting-software integrations.

## Project structure

```
backend/    FastAPI app: API routes, pydantic models, pipeline services
frontend/   Streamlit app: SME upload/dashboard/passport + lender comparison view,
            calling the backend through an HTTP adapter with mock-data fallback
scripts/    synthetic data generation
CLAUDE.md   instructions for the AI coding agent
```
